Binder API

The Binder API is in development, and will release soon after the game. The API (entitled Knapsack) is fully integrated, and will work fully with the obfuscated code. The Binder source will come obfuscated to prevent any copyright-infringement, but the API will add a clean, easy to use face on the messy code. For example in the Binder code (example class) the chunk class will be class Z, to render it we would call:

Z a = new Z(0, 0, 0);
a.d();

But using Knapsack (which does all the nitty-gritty stuff) it would simply be:

Chunk c = new Chunk(0, 0, 0);
c.render();

Easy huh?

Plugin System

Another improvement to the game code we made is that Knapsack is able to load plugins. Plugins are little JAR files that contain new classes to expand Binder. The classes can be new worlds, new monsters, new blocks, new shapes, new textures, you name it! Each plugin is activated by the PluginLoader class. In order for a plugin to be activated, it must have an entry in the Plugins.json that follows the default formatting. So, in order to have your plugin activated on runtime simply add these lines to the Plugins.json file:

"ExamplePlugin": {
  "main": "me.wxwsk8er.ExamplePlugin.ExampleClass",
  "name": "ExamplePlugin",
  "version": "0.0.1",
  "author": "wxwsk8er",
  "req_knapsack_v": "0.0.1"
}

Notice that these are JSON formatted, this is required! Now, if you don’t understand any of that let me explain. Hopefully you are a little used to with the JSON language (or YAML) and can see the structure. It follows this structure:

  • [Plugin Entry Name]
    • [Plugin main class (extends JavaPlugin)]
    • [Plugin name]
    • [Plugin version]
    • [Plugin (lead) author]
    • [Version of Knapsack it was created for]
  • …

Each plugin must have ONE class that extends JavaPlugin, this is the class path that you use for the main attribute in your entry. This class will have a dedicated GID, configuration module, enable and disable methods, and a full event listener system. You can see the full list of items by typing this. in eclipse, all fields should showup.

Configuration Module

Knapsack ships with a built-in configuration module already setup to work with JSON configuration files. Full get(), isSet(), and save() methods are included. Knapsack not only provides a working JSON configuration system, but even better [other than smiles]! Knapsack comes with abstract interfaces allowing you to create your own configuration modules. Infact, you could even make your own configuration language and use it!

For starters, every Configuration modules starts as a replica of the ConfigurationSection class. If all you want to do is use your module as a memory configuration, just implement the Configuration class. If you want to have full file access (save(), read(), write(), etc) you will need to extend the FileConfiguration class. This class allows the creation of file configuration modules that can be recognized universally. The extends keyword masks the class’s data type to be FileConfiguration, and allows the interfacing methods to be used universally.
